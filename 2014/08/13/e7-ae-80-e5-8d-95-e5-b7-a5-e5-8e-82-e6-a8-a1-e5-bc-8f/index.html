首先是接口,面向接口编程是java设计原则之一,面向接口编程好处我就不千篇一律的啰嗦一大堆了.一句话”接口是系统可插拔的保证“

接口的功能是做什么:

接口是用来对外提供功能调用或者提供服务,但是却是隐藏了其的实现细节给它的调用者.调用者只知道业务功能接口,调用接口提供的功能而完成自己的事务.

接口的思想是:

“封装隔离”是接口的思想,这里的封装是对被隔离体行为的封装,隔离就是外部不知道内部的实现只是会调用.

在java程序设计里面非常讲究层的划分和模块的划分,通常按照三层划分:表现层+逻辑层+数据层.在这些层之间的通讯就是通过接口来实现.有人定义接口就直接就是”层之间通讯的工具”,在每一个层里面又有很多小的模块,每个模块对外都是一个整体,所以也应该对外提供相应的接口,模块之间也应该是接口调用 “接口是被其隔离部分的外观和代表”

在一般的编程中我们一般是定义了功能的抽象来对类进行约束,然后提供实现,最后供给调用,不妨接下来的介绍中使用:

Api:代表接口

Impl:代表实现,ImplA代表一个A实现,类推

Client:代表外部调用

一般模式:

client代码 Api api = new Impl(); api.method()…..这样存在几个问题,首先是接口是”隔离封装”的代名词,此处虽然是用了接口编程但是没有体现最重要的”隔离”,就是说,客户端(调用者)不但要知道这个接口,还要知道这个实现类,如果实现类很多的化就….总之隔离要求的是client也不能知道Impl,Impl是隔离体

解决办法:简单工厂类,下文就用factory代表

我们将new Impl()这个动作交个factory去处理,client只需要返回一个Api即可,这里又存在一个问题就是如何知道要选择那个实现类呢

我们可以通过传入参数去告诉

public class Factory{ public static Api createApi(args){ if(args == A){ return new ImplA(); } if(......) //选择条件 } }
关于Factory类的一些建议:
1.想要获取那个接口,一般就是create+接口名或者get+接口名;
2.一般工厂类不希望也没有必要去实例化,所以一般就是静态方法;
3.工厂一般作为工具类使用;
4.一般一个工厂类的命名是以模块名+Factory;

这样的化就可以将实现类的选择交个工厂去处理,”简单工厂类的核心就是选择接口的实现类”

大家或许发现了一个问题,就是这样的程序还有问题:
1.client还是必须知道参数的含义,才能选择相应的接口,还是不是很友善
2.要是增加了选择的条件args,或者改变或者增加实现类,这样的话还要去手动修改factory类,不方便,比如我们希望在添加实现类的时候不在修改factory类

为了解决上面的问题,一个简单的方法是使用配置文件,当有了新的实现类的时候只要在配置文件上面直接配置就可以了在简单工厂里面可以通过反射或者IOC/DI去解决,以反射为例,伪代码如下:
public class Factory{ //Factory.properties 里面提供了实现类的配置 如 ImplClass = ImplC public static Api createApi(){ InputStream in = Factory.class.getResourceAsStream("Factory.properties") Properties prop = new Propertier(); prop.load(in); //省了很多代码 return (Api)Class.forName(p.getProperty(ImplClass)).newInstance();

}

}

简单工厂优点:

1.帮助封装,隔离更加彻底,真正的面向接口编程,外部什么都不用知道(一般功能的化是肯定知道了,可以这样理解接口全是按照client要求做的,所以client能调用,但是其余就不应该由client操心了)
2.实现解耦,客户端和具体实现类的解耦

何时选择简单工厂建议:

1.完全封装,只能”傻瓜的访问自己所需的功能”,其余事情交factory去管理

2.如果想要把对外创建对象的职责集中管理和控制,可以选择工厂一个简单工厂可以创建很多不相关的对象,由于是一个”中间件”
可以在创建对象的时候做很多工作和”代理”差不多,比如创建单例,权限创建,格式化等(配合其他模式一起使用)….总之对象的集中管理,想咋做就咋做呗
注意:一般一个工厂类的实例化类一般是在一个模块里面的类,不能太多,否则可能会职责冗余.
